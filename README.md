## Quick Start (Developer)

Краткая инструкция для быстрой локальной разработки без Docker/Podman.

Вариант A — быстро (SQLite, рекомендуется для прототипа):

```bash
cd server
npm install
# создайте .env с переменными (пример ниже)
cat > .env <<'EOF'
DATABASE_PROVIDER=sqlite
DATABASE_URL="file:./dev.db"
JWT_SECRET="change_this_secret"
PORT=5000
EOF

npm run prisma:generate
npx prisma db push
npm run dev
```

Вариант B — полноценно (PostgreSQL):

1. Установите PostgreSQL (или используйте managed DB).
2. Создайте пользователя и базу, затем настройте `DATABASE_URL` в `.env`.

Пример (Ubuntu):
```bash
sudo apt update && sudo apt install -y postgresql
sudo -u postgres createuser -P marketplace_user
sudo -u postgres createdb -O marketplace_user marketplace_db
# в server/.env: DATABASE_PROVIDER=postgresql
# DATABASE_URL="postgresql://marketplace_user:password@localhost:5432/marketplace_db?schema=public"
cd server
npm install
npm run prisma:generate
npx prisma migrate dev
npm run dev
```

Примечания:
- Для тестов и простого локального прототипа используйте SQLite.
- Для интеграционных тестов и продакшена используйте PostgreSQL и миграции.

## 1. Общие положения

### 1.1 Назначение документа
Настоящий документ определяет технические требования к разработке платформы-маркетплейса веб-сервисов с моделью оплаты по подписке и автоматическим развертыванием экземпляров сервисов в изолированных Docker-контейнерах.

### 1.2 Цели создания системы
- Создание платформы для продажи доступа к веб-сервисам по подписке
- Автоматизация процесса развертывания персональных экземпляров сервисов
- Обеспечение полной изоляции данных между клиентами
- Предоставление единого личного кабинета для управления всеми подписками
- Интеграция с российской платежной системой ЮKassa

### 1.3 Основные термины
| Термин | Определение |
|--------|-------------|
| **Платформа** | Основной сайт (оркестратор), управляющий пользователями, подписками и развертыванием |
| **Сервис** | Веб-приложение, доступное для покупки (Канбан-доска, CRM, AI-инструмент и т.д.) |
| **Тенант (Tenant)** | Персональный экземпляр сервиса, развернутый для конкретного клиента |
| **Подписка** | Запись о доступе пользователя к конкретному сервису с определенным тарифом |
| **Оркестратор** | Основное приложение, управляющее всей платформой |

## 2. Описание проекта

### 2.1 Концепция
Платформа представляет собой маркетплейс, где пользователи могут:
1. Просматривать каталог веб-сервисов
2. Выбирать тарифный план и оплачивать подписку
3. Мгновенно получать персональный экземпляр выбранного сервиса
4. Получать доступ по уникальному поддомену
5. Управлять всеми подписками в едином личном кабинете

### 2.2 Основные сценарии использования

#### Сценарий 1: Регистрация и первый вход
1. Пользователь заходит на главную страницу платформы
2. Регистрируется по email/паролю или через соцсети
3. Подтверждает email
4. Входит в личный кабинет

#### Сценарий 2: Покупка подписки на сервис
1. Пользователь просматривает каталог сервисов
2. Выбирает интересующий сервис (например, Канбан-доску)
3. Изучает тарифы (Basic, Pro, Enterprise)
4. Выбирает тариф и период (месяц/год)
5. Указывает название компании (для поддомена)
6. Переходит к оплате через ЮKassa
7. Вводит данные карты на стороне ЮKassa
8. После успешной оплаты перенаправляется на платформу
9. Получает email с ссылкой на развернутый сервис
10. Переходит по ссылке и начинает работу

#### Сценарий 3: Управление подписками
1. Пользователь заходит в личный кабинет
2. Видит список всех активных подписок
3. Для каждой подписки доступны:
   - Ссылка на сервис
   - Текущий тариф
   - Дата окончания периода
   - Кнопка отмены подписки
   - Кнопка смены тарифа

#### Сценарий 4: Автоматическое продление
1. Система ежедневно проверяет подписки с истекающим сроком
2. За 3 дня до окончания отправляет email-уведомление
3. В день окончания пытается списать средства с сохраненной карты
4. При успехе - продлевает период, отправляет уведомление
5. При неудаче - меняет статус на "past_due", отправляет уведомление
6. Через 5 дней неудачных попыток - приостанавливает контейнер

#### Сценарий 5: Отмена подписки
1. Пользователь нажимает "Отменить подписку"
2. Подтверждает действие
3. Подписка помечается как "cancel_at_period_end"
4. Контейнер продолжает работу до конца оплаченного периода
5. По окончании периода контейнер останавливается
6. Данные хранятся 30 дней (возможно восстановление)
7. Через 30 дней контейнер и данные удаляются

#### Сценарий 6: Добавление нового сервиса (админ)
1. Администратор заходит в админ-панель
2. Заполняет форму:
   - Название сервиса
   - Описание
   - Логотип
   - Категория
   - Docker-образ
   - Тарифы с ценами и лимитами
3. Сохраняет сервис
4. Сервис появляется в каталоге

## 3. Функциональные требования

### 3.1 Пользовательская часть (Frontend)

#### Главная страница
- Шапка с навигацией (Каталог, Цены, Вход/Регистрация)
- Промо-блок с описанием платформы
- Список популярных сервисов
- Призыв к действию (CTA)

#### Каталог сервисов
- Сетка сервисов с фильтрацией по категориям
- Поиск по названию
- Карточка сервиса: логотип, название, краткое описание, цена от
- Страница сервиса:
  - Полное описание
  - Скриншоты/видео
  - Список ключевых функций
  - Сравнение тарифов (таблица)
  - Кнопка "Начать" (выбор тарифа)

#### Аутентификация
- Регистрация по email/пароль
- Вход по email/пароль
- Восстановление пароля (через email)
- Подтверждение email
- Социальные сети (опционально: Google, VK)

#### Личный кабинет
- Дашборд с общей информацией
- Список активных подписок:
  - Название сервиса
  - Тариф
  - Дата окончания
  - Ссылка на сервис
  - Кнопки управления
- История платежей
- Настройки профиля (смена пароля, email, имени)
- Настройки компании (название, поддомен по умолчанию)

#### Процесс покупки
- Выбор тарифа и периода
- Форма с названием компании (для поддомена)
- Переход на оплату (редирект на ЮKassa)
- Страница успешной оплаты
- Страница ошибки оплаты

### 3.2 Административная часть (Admin Panel)

#### Управление сервисами
- CRUD для сервисов
- Загрузка логотипов и скриншотов
- Настройка Docker-образа для каждого сервиса
- Настройка тарифов (CRUD для планов)

#### Управление пользователями
- Список всех пользователей
- Просмотр деталей пользователя
- Блокировка/разблокировка
- Просмотр подписок пользователя

#### Управление подписками
- Список всех подписок
- Фильтры по статусу, сервису, дате
- Ручное создание/редактирование подписки
- Принудительная остановка контейнера

#### Мониторинг
- Дашборд с метриками:
  - Количество активных подписок
  - Количество запущенных контейнеров
  - Загрузка CPU/RAM по серверам
  - Доход (MRR)
- Список контейнеров с их статусами
- Логи контейнеров (просмотр)
- Алерты о проблемах

#### Платежи и финансы
- Список всех платежей
- Экспорт в Excel/CSV
- Статистика по платежам
- Возвраты (refunds)

### 4.2 Компоненты системы

#### Оркестратор (Main API)
- **Язык:** Node.js
- **Фреймворк:** Express.js
- **Порт:** 5000
- **Функции:**
  - Управление пользователями (CRUD, аутентификация)
  - Управление каталогом сервисов (CRUD)
  - Управление подписками (создание, продление, отмена)
  - Интеграция с ЮKassa (платежи, вебхуки)
  - Управление Docker-контейнерами (запуск, остановка, удаление)
  - Отправка email-уведомлений
  - API для фронтенда

#### Основная база данных
- **СУБД:** PostgreSQL
- **Схема:** Описана в разделе 6

#### Reverse Proxy
- **Инструмент:** Traefik
- **Функции:**
  - Маршрутизация трафика
  - Автоматическое получение SSL-сертификатов
  - Балансировка нагрузки (для масштабирования)

#### Контейнеры клиентов
- **Технология:** Docker
- **Структура каждого контейнера:**
  - Node.js + Express (бэкенд)
  - React (фронтенд)
  - PostgreSQL (локальная БД)
  - Переменные окружения с лимитами

- Создание одноразовых платежей
- Создание рекуррентных платежей (подписки)

- Обработка вебхуков (уведомлений об изменении статуса)

#### События вебхуков для обработки

| payment.waiting_for_capture | Ожидать подтверждения (для двухстадийных) |
| refund.succeeded | Обновить статус платежа, деактивировать подписку |

- Проверка IP-адресов отправителей (белый список ЮKassa)
- Хранение чувствительных данных только на стороне ЮKassa
- Использование Idempotence-ключей для защиты от дублирования

## 9. Требования к безопасности

### 9.1 Аутентификация и авторизация
- Хранение паролей в хешированном виде (bcrypt)
- JWT токены с ограниченным сроком действия (access + refresh)
- Защита от brute-force (rate limiting)
- Подтверждение email перед активацией
- Ролевая модель (user/admin)

### 9.2 Защита данных
- Все соединения по HTTPS (SSL/TLS)
- Изоляция контейнеров (разные сети, лимиты ресурсов)
- Санитизация входных данных (XSS, SQL injection)
- CORS настроен только для доверенных доменов
- Секреты в переменных окружения (не в коде)

### 9.3 Безопасность платежей
- Никакие платежные данные не хранятся на сервере
- Использование официального SDK ЮKassa
- Проверка вебхуков по IP
- Idempotence-ключи для защиты от дублей

### 9.4 Безопасность Docker
- Контейнеры запускаются от непривилегированного пользователя
- Лимиты ресурсов (CPU, RAM) для каждого контейнера
- Read-only root filesystem где возможно
- Регулярное обновление базовых образов

### 9.5 Резервное копирование
- Ежедневный бэкап основной БД
- Бэкап данных контейнеров (volumes)
- Хранение бэкапов 30 дней
- Тестирование восстановления раз в месяц

## Локальный запуск (Podman)

Используйте Podman вместо Docker, команды остаются почти идентичными.
Если у вас нет `podman-compose`, установите его через `pip`:

```bash
pip install podman-compose
```

или воспользуйтесь встроенным Makefile:

```bash
cp .env.example .env             # скорректируйте значения
make up                          # поднимет Postgres + оркестратор
make logs                        # вывод логов
docker-compose down или make down
```

`docker-compose` тоже работает — Makefile автоматически выберет
доступную утилиту.
